Memoria con Recuperación para el Agente Local No enviar todo el historial: ventana reciente fija + recuperación de lo relacionado + búsqueda agéntica

Idea de Rhay · estructurado por Claude · 21 may 2026

1El problema

Enviar todo el historial en cada turno no escala: crece el costo por token, se llena la ventana de contexto y el modelo se distrae con ruido viejo. La meta es enviar solo lo necesario y dejar que el agente recupere el resto bajo demanda desde un almacén persistente (SQLite), no desde la pila en memoria.

2Principios de diseño

🔒 Inicio fijo

claude.md / agent.md siempre presentes. Define identidad, reglas y herramientas.

🪟 Ventana reciente

Los últimos N turnos van siempre. Es lo que salva la continuidad ("eso", "el de antes", "hazlo igual").

🎯 Recuperado

Top-K mensajes/​resúmenes relacionados con el turno actual, traídos por similitud + recencia + grafo.

Regla de oro: el contexto enviado = inicio fijo + ventana reciente + recuperados. Nada más. Si el modelo necesita más, lo pide con una herramienta.

3El ciclo, por turno

se envía al LLM →
① system: claude.md / agent.md fijo
② últimos N turnos (working memory) siempre
③ top-K recuperados de SQLite dinámico
↓   LLM genera respuesta   ↓
si falta info →
④ tool buscar_memoria(query) → memoria ampliada → preguntar al usuario
↓   se guarda user-msg + respuesta en SQLite (con embedding)   ↓
⑤ reinicia el ciclo (la ventana avanza, lo viejo se resume)

4Esquema SQLite

-- Mensajes: la fuente de verdad de la conversación
CREATE TABLE messages (
  id         INTEGER PRIMARY KEY,
  session_id TEXT,
  rol        TEXT,            -- user | assistant | system
  texto      TEXT,
  ts         INTEGER,         -- epoch
  tokens     INTEGER,
  embedding  BLOB,            -- vector del texto
  bloque_id  INTEGER           -- bloque resumido al que pertenece
);

-- Grafo de relaciones (FASE 2, opcional)
CREATE TABLE relations (
  from_id INTEGER, to_id INTEGER,
  tipo    TEXT,               -- responde_a | continua | referencia | tema
  peso    REAL,
  PRIMARY KEY (from_id, to_id, tipo)
);

-- Resúmenes rodantes: comprimen tramos viejos para no recuperar ruido
CREATE TABLE bloques (
  id INTEGER PRIMARY KEY, session_id TEXT,
  desde_ts INTEGER, hasta_ts INTEGER,
  resumen TEXT, embedding BLOB
);

-- Búsqueda vectorial nativa con sqlite-vec (local, sin servidor)
CREATE VIRTUAL TABLE vec_messages USING vec0(embedding float[768]);

5Pseudocódigo del turno

def on_user_message(m):
    emb = embed(m.texto)                          # modelo de embeddings local
    save(messages, m, emb)                       # persistir SIEMPRE
    relate(m, recientes)                         # opcional: actualizar grafo

    ctx  = [system(claude_md)]                   # ① inicio fijo
    ctx += ultimos_N_turnos()                    # ② working memory (siempre)
    ctx += retrieval(emb, excluir=ctx, k=8)     # ③ relacionados

    resp = llm(ctx, tools=[buscar_memoria, ...])
    while resp.tool == "buscar_memoria":        # ④ recuperación agéntica
        hits = retrieval(embed(resp.query), k=8)
        if not hits: hits = memoria_ampliada(resp.query)   # MEMORY.md / memoria.json
        if not hits: resp = preguntar_al_usuario(resp.query) # escalado final
        resp = llm(ctx + hits, ...)

    save(messages, resp, embed(resp.texto))
    maybe_resumir_bloque()                       # ⑤ resúmenes rodantes

6El componente de recuperación (lo que decide qué es "relacionado")

Solo grafo no basta; solo similitud tampoco. Score híbrido:

score(c) = w₁·cos_sim(emb_actual, c.emb)   # de qué habla
         + w₂·recencia(c.ts)               # qué tan reciente
         + w₃·grafo_hops(actual, c)        # conexión causal (fase 2)

# top-K por score, con umbral mínimo de similitud para no traer basura

RecenciaSesgar a lo reciente sin botar lo viejo relevantedecay temporal sobre ts

7Fases sugeridas

FASE 1 El 80% del valor

Ventana reciente fija + retrieval por embeddings + tool buscar_memoria + resúmenes rodantes.

FASE 2 El grafo

Tabla relations y score con saltos. Solo si la recuperación pura se queda corta.

OBSERVAR Trazabilidad

Loggear "qué se recuperó y por qué" en cada turno. Imprescindible para debuggear el agente.

8Riesgos a tener en cuenta

  • crítico  Botar los turnos recientes rompe la coherencia. La ventana fija es innegociable.
  • latencia  Cada turno hace embedding + búsqueda. Con sqlite-vec local es asumible, pero medir.
  • modelo local  El retrieval agéntico exige buen tool-calling; modelos pequeños fallan ahí (ver evaluaciones de compatibilidad LLM).
  • ruido  Sin resúmenes rodantes, recuperas mensajes crudos viejos irrelevantes. Resumir bloques lo arregla.

Veredicto

La dirección es la correcta y es hacia donde van los agentes serios (MemGPT/Letta, mem0, Zep/Graphiti). El 80% del valor lo da la Fase 1: ventana reciente fija + retrieval por embeddings + tool de búsqueda agéntica + resúmenes rodantes. El grafo de relaciones es un excelente 20% extra — pero déjalo para la Fase 2, cuando ya tengas el ciclo andando y veas qué se queda corto.

9Decisiones abiertas (para ti)

Embeddings: ¿modelo local (nomic-embed/bge vía Ollama o sentence-transformers) o un endpoint? Para tu setup local, local.
Tamaño de ventana N: ¿últimos 6–10 turnos? Se calibra con pruebas.
Cuándo resumir: ¿cada X turnos o cuando un bloque supera Y tokens?
Integración: ¿esto reemplaza el manejo de historial actual del agente, o convive como capa nueva sobre memoria.json / MEMORY.md (que serían la "memoria ampliada")?
Próximo paso propuesto: montar un POC mínimo en SQLite + sqlite-vec + embeddings locales que implemente el ciclo de la sección 5 (sin grafo), para verlo andar con conversaciones reales del agente.
