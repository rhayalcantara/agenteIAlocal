ALo que ya funciona

Probado end-to-end en localhost. POST con token devuelve id, GET drena el inbox, segunda llamada vuelve vacía.

  • server.py — FastAPI con POST /inbox, GET /poll, GET /health. Auth header X-Bridge-Token.
  • client.py — helpers send(), poll(), health() reutilizables.
  • requirements.txt — fastapi · uvicorn · pydantic · python-dotenv · requests.
  • .env — BRIDGE_TOKEN aleatorio (32 chars), BRIDGE_PORT, BRIDGE_NODE.
  • .gitignore — bridge_inbox.json excluido.
  • README.md — instalación + uso + cloudflared.

BCómo usarlo desde Claude

Dos verbos: enviar al inbox del otro lado, leer el propio.

Enviar CLAUDE → OTRO

from bridge.client import send

send(
    "git status del repo",
    to_url="https://claude-ranger.xxx",
    from_="claude-local"
)
# → {'ok': True, 'id': 1778..., 'queued': 1}

Leer MIS MENSAJES

from bridge.client import poll

for m in poll():
    print(m["from"], m["text"])
# consume=true por default — vacía el inbox

CLo que falta

Tres pasos para llegar al setup completo bidireccional cross-network.

  1. Cloudflared instalado + túnel nombrado

    Bajar binario, cloudflared tunnel login, cloudflared tunnel create claude-local, route DNS al dominio (o trycloudflare ephemeral para test rápido).

  2. Servicio persistente

    NSSM o Tarea Programada al boot — server FastAPI + cloudflared. Sobrevive reinicio.

  3. Replicar en Ranger

    git pull → pip install → token compartido en su .env → cloudflared con nombre claude-ranger. Cada lado escribe BRIDGE_REMOTE_URL apuntando al otro.
